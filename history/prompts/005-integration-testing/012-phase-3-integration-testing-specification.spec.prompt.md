`--duration`: seconds (default: 86400 = 24 hours) - `--rate`: messages per hour (default: 50) ### 6.2 `SimulationMetrics` Class | Attribute | Type | Purpose | |-----------|------|---------| | `start_time` | datetime | Simulation start | | `total_sent` | int | All messages attempted | | `total_success` | int | HTTP 200 responses | | `total_failed` | int | Non-200 or exceptions | | `latencies` | list[float] | Millisecond response times | | `by_channel` | dict | Count per channel | | `errors` | list[dict] | Error log (channel, error, time) | **Methods:** - `record_success(channel, latency_ms)` — increment counters - `record_failure(channel, error)` — log failure - `get_p95_latency()` — sort latencies, take 95th percentile index - `get_uptime_percent()` — `(success / sent) * 100` - `print_summary()` — print table + validation checks ### 6.3 Test Messages **8 realistic web form messages** covering categories: technical (4), billing (1), bug_report (1), general (1), feedback (1). Each message has: - `subject`: Short descriptive subject (5–50 chars) - `category`: One of the 5 valid categories - `message`: Realistic customer message (50–300 chars) ### 6.4 `send_web_form(client)` Function 1. Pick random message from `TEST_MESSAGES["web_form"]` 2. Pick random name from `NAMES` list (10 realistic names) 3. Generate unique ID: `int(time.time() * 1000) % 100000` 4. POST to `/support/submit` with `timeout=30.0` 5. Measure latency in milliseconds 6. On 200: `metrics.record_success("web_form", latency_ms)` 7. On non-200: `metrics.record_failure("web_form", f"HTTP {status_code}")` 8. On exception: `metrics.record_failure("web_form", str(e))` ### 6.5 `run_simulation(duration_seconds, messages_per_hour)` Function **Interval calculation:** `interval = 3600 / messages_per_hour` (seconds between messages) **Main loop:** 1. Pre-flight: `check_health(client)` → abort if not healthy 2. Log "simulation started" 3. While `time.time() < end_time`: a. `await send_web_form(client)` b. Every 10 messages: `await check_health(client)` → log if unhealthy c. Every 100 messages: log progress (total sent, elapsed hours, P95) d. `await asyncio.sleep(interval)` 4. `metrics.print_summary()` 5. Save to `simulation_results.json` **`simulation_results.json` schema:** ```json { "start_time": "ISO8601", "end_time": "ISO8601", "total_sent": 400, "total_success": 399, "total_failed": 1, "p95_latency_ms": 1240.5, "uptime_percent": 99.75, "by_channel": {"web_form": 400}, "errors": [{"channel": "web_form", "error": "...", "time": "..."}] } ``` ### 6.6 Simulation Pass Criteria | Metric | Minimum Target | How Verified | |--------|---------------|-------------| | Web form submissions | ≥ 100 | `by_channel["web_form"] >= 100` | | Uptime | > 99.9% | `uptime_percent >= 99.9` | | P95 latency | < 3000ms | `p95_latency_ms < 3000` | | Failure rate | < 1% | `total_failed / total_sent < 0.01` | --- ## 7. Returns list of pod names. #### `kill_pod(pod_name) → bool` ``` kubectl delete pod {pod_name} -n customer-success-fte --grace-period=0 --force ``` Returns True if returncode == 0. #### `wait_for_recovery(timeout_seconds=120) → bool` Loop every 5 seconds until `GET /health` returns 200 with `status=="healthy"`. Timeout = 120 seconds. Returns True if recovered, False if timed out. #### `verify_no_message_loss(client, ticket_ids) → bool` For each ticket_id: `GET /support/ticket/{ticket_id}`. All must return 200. Returns True if ALL tickets survive. ### 7.3 `run_chaos_cycle(client, cycle_number)` — Exact Steps | Step | Action | On Failure | |------|--------|-----------| | 1 | Submit 5 test tickets BEFORE chaos | Log error, continue | | 2 | `get_running_pods("api")` | Log warning if no pods | | 3 | `kill_pod(random.choice(api_pods))` | Log warning if kill fails | | 4 | `wait_for_recovery(timeout=120)` | Return False if timeout | | 5 | Submit 3 more tickets AFTER recovery | Log error | | 6 | `verify_no_message_loss(client, all_ticket_ids)` | Return False if any missing | | 7 | Log result: PASSED or FAILED | — | Returns: `True` if both recovery and zero message loss confirmed. **Docker Compose fallback** (if no K8s): When no pods found, test recovery by just submitting tickets and checking they survive. ### 7.4 `chaos_results.json" Schema ```json { "cycles": 3, "passed": 3, "failed": 0, "timestamp": "ISO8601" } ``` ### 7.5 Chaos Test Pass Criteria | Metric | Target | |--------|--------| | Recovery time | < 120 seconds per kill | | Message loss | Zero — all pre/post tickets accessible | | Cycles passed | All cycles pass | | System stable | Health check returns healthy after each kill | --- ## 8. MONITORING & METRICS SPECIFICATION ### 8.1 File: `production/monitoring/metrics_dashboard.py` **Purpose:** Real-time console dashboard reading from PostgreSQL every 60 seconds. **Run command:** `python production/monitoring/metrics_dashboard.py` ### 8.2 SQL Queries Required #### Query 1: Channel breakdown (last 24h) ```sql SELECT initial_channel as channel, COUNT(*) as total, COUNT(*) FILTER (WHERE status = 'resolved') as resolved, COUNT(*) FILTER (WHERE status = 'escalated') as escalated, COUNT(*) FILTER (WHERE status = 'active') as active, ROUND(AVG(sentiment_score)::numeric, 2) as avg_sentiment FROM conversations WHERE started_at > NOW() - INTERVAL '24 hours' GROUP BY initial_channel ``` #### Query 2: Latency percentiles (from agent_metrics) ```sql SELECT channel, ROUND(PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY metric_value)::numeric, 0) as p95_ms, ROUND(AVG(metric_value)::numeric, 0) as avg_ms, COUNT(*) as sample_count FROM agent_metrics WHERE metric_name = 'latency_ms' AND recorded_at > NOW() - INTERVAL '24 hours' GROUP BY channel ``` #### Query 3: Overall stats (last 24h) ```sql SELECT COUNT(*) as total_conversations, COUNT(*) FILTER (WHERE status = 'resolved') as total_resolved, COUNT(*) FILTER (WHERE status = 'escalated') as total_escalated, ROUND(AVG(sentiment_score)::numeric, 2) as overall_sentiment FROM conversations WHERE started_at > NOW() - INTERVAL '24 hours' ``` #### Query 4: Message count (last 24h) ```sql SELECT COUNT(*) as total_messages FROM messages WHERE created_at > NOW() - INTERVAL '24 hours' ``` ### 8.3 Dashboard Output Format ``` ================================================================= CUSTOMER SUCCESS FTE — LIVE METRICS 2024-01-15T10:30:00Z ================================================================= 📊 LAST 24 HOURS (OVERALL) Total conversations: 142 Resolved: 98 (69.0%) Escalated: 28 (19.7%) | Target: <25% Avg sentiment: 0.65 Total messages: 284 📡 BY CHANNEL EMAIL | Total: 45 | Resolved: 30 | Escalated: 8 (17%) | Sentiment: 0.71 WHATSAPP | Total: 37 | Resolved: 25 | Escalated: 9 (24%) | Sentiment: 0.58 WEB_FORM | Total: 60 | Resolved: 43 | Escalated: 11 (18%) | Sentiment: 0.68 ⚡ LATENCY (P95) EMAIL | P95: 1245ms ✅ | Avg: 890ms | Samples: 45 WHATSAPP | P95: 987ms ✅ | Avg: 734ms | Samples: 37 WEB_FORM | P95: 1567ms ✅ | Avg: 1120ms | Samples: 60 ✅ SLA STATUS Escalation rate: ✅ PASS (19.7% < 25%) P95 Latency: ✅ PASS (1567ms < 3000ms) ================================================================= ``` ### 8.4 SLA Validation Logic | SLA | Formula | Pass Condition | |-----|---------|---------------| | Escalation rate | `escalated / max(1, total) * 100` | < 25% | | P95 latency | `max(p95_ms for all channels)` | < 3000ms | --- ## 9.
